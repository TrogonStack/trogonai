# Key Management

This page describes the intended design recorded in
[ADR#0023](../adr/0023-secret-management-and-key-custody-direction.md)
(accepted) and
[ADR#0030](../adr/0030-customer-controlled-key-backend-routing.md) and
[ADR#0033](../adr/0033-two-tier-key-custody-product-model.md) (both draft).
None of it is implemented yet. It is the model the implementation must
satisfy, not current behavior.

## Service topology

A platform secrets service is the only process that holds a provider client
or provider credential. Callers never talk to a provider directly: they hold
opaque references and resolve them through the secrets service on demand
([ADR#0023](../adr/0023-secret-management-and-key-custody-direction.md)
Decision 3).

Sensitive values move on one path only: authenticated, encrypted
[NATS](../glossary/nats) core request/reply between an authorized caller
and the secrets service. A DEK crosses the service boundary this way, into
the service for `wrap_dek` and back to the caller after `unwrap_dek`, and
never by any other route. It never enters
[JetStream](../glossary/jetstream), another durable transport, a log, or a
trace
([ADR#0030](../adr/0030-customer-controlled-key-backend-routing.md)
Decision 2).

JetStream carries only the metadata that surrounds key material, never the
material itself: rotation, revocation, and invalidation events, plus the
correlation metadata that ties a business-context audit record to the
provider's own audit log
([ADR#0023](../adr/0023-secret-management-and-key-custody-direction.md)
Decision 3 and Decision 6).

## Ports

Business code sees two typed ports, never a generic provider interface or a
provider SDK type.

**SecretStore** stores and resolves retrievable credentials. A caller holds a
`SecretRef`, an opaque handle the service resolves internally to
[tenant](../glossary/tenant), vault, and key. The value itself is
retrievable, and that is the point: tokens, webhook signing secrets, and API
credentials belong here.

**KeyManagement** performs cryptographic operations against a `KeyRef` with
an explicit key purpose. The key material is never retrieved by a caller.
This page covers only the `EnvelopeWrapping` capability of `KeyManagement`,
the wrap and unwrap contract that
[ADR#0030](../adr/0030-customer-controlled-key-backend-routing.md) defines
([ADR#0023](../adr/0023-secret-management-and-key-custody-direction.md)
Decision 2).

## Concepts

### Dek

A `Dek` (data-encryption key) is generated by the authorized encryption
caller from a cryptographically secure random source. Its size is fixed by a
versioned envelope-cipher suite, which is an open decision (see
[Open decisions](#open-decisions)); the initial conformance suite targets a
32-byte `Dek` supported by all three backends. Every application-held usable
copy lives in a single-owner secret wrapper that zeroizes its buffer on every
operation exit path, success, error, cancellation, or retry. A `Dek` is never
copied into application state, a reusable cache, a log, a trace, or a durable
transport.

### WrappedDek

A `WrappedDek` is a platform-owned, versioned envelope. It contains provider
ciphertext behind an adapter-private payload codec, and that ciphertext is
opaque to everything outside its own adapter: no provider marker or format
appears at the boundary. The data-owning subsystem persists the `KeyRef`, the
`WrappedDek`, the binding version, and the payload ciphertext together, but
it never parses the `WrappedDek`. Raw
provider ciphertext carried as an opaque payload inside a `WrappedDek` is the
only exception to the rule that provider artifacts never cross the service
boundary
([ADR#0030](../adr/0030-customer-controlled-key-backend-routing.md)
Decision 2).

### KeyRef

A `KeyRef` is an opaque, service-resolved reference to a key-encryption key.
It is bound immutably to exactly one `KeyBackendId` and one native provider
locator; the binding cannot change in place. A data-plane caller never
selects a backend, supplies a provider resource name, or parses a `KeyRef`.
Moving ciphertext to a different backend always creates a new `KeyRef`
through the rewrap migration described under
[Operations](#operations), never a change to an existing one
([ADR#0030](../adr/0030-customer-controlled-key-backend-routing.md)
Decision 1 and Decision 4).

### AuthenticatedBinding

An `AuthenticatedBinding` is a typed, versioned, deterministic, non-secret
value derived from the authenticated tenant, the authoritative resource
identity, the key purpose, and the binding schema version. It never derives
tenant or resource identity from a caller-supplied envelope field. The
identical binding is required to unwrap a `Dek`, which prevents a
`WrappedDek` from being moved to a different tenant, resource, or purpose.

Each adapter maps an `AuthenticatedBinding` onto the provider's own
authenticated-data mechanism:

- OpenBao Transit `associated_data` on a supported AEAD key type. OpenBao
  `context`, which controls key derivation, is not the common binding.
- AWS KMS `EncryptionContext`, which is case-sensitive and appears in
  plaintext in AWS audit logs. The binding therefore contains no secrets.
- Google Cloud KMS `additionalAuthenticatedData`.

([ADR#0030](../adr/0030-customer-controlled-key-backend-routing.md)
Decision 2.)

### KeyBackendRegistration and KeyBackendId

A `KeyBackendRegistration` is durable, tenant-scoped product state. It owns
connection, authentication, and trust configuration for one backend, but no
key locator. It has an opaque `KeyBackendId`, a typed provider variant,
ownership, lifecycle status, provider access parameters, and a non-secret
authentication profile. Raw credentials never appear in configuration, an
event, or the registration itself; an unavoidable secret is represented by a
`CredentialRef` and stays behind `SecretStore`.

`KeyBackendId` identifies an immutable provider account, project, or OpenBao
security boundary. A credential, trust root, authorization principal, or
OpenBao endpoint update may create a new registration revision under the
same `KeyBackendId` only when validation proves continuity with the same
boundary and its already-bound keys. An account, project, or unproven
endpoint change requires a new `KeyBackendId` instead. Lifecycle and health
fields may change without changing identity
([ADR#0030](../adr/0030-customer-controlled-key-backend-routing.md)
Decision 3).

### Purpose

A key purpose is part of both resolution and binding. On `wrap_dek`, the
service resolves a `KeyRef` from the authenticated tenant, the purpose, and
tenant policy. On `unwrap_dek` and inside every `AuthenticatedBinding`, the
purpose must match what was bound at wrap time, or the operation fails
closed.

## Operations

### wrap_dek

```text
wrap_dek(KeyRef, Dek, AuthenticatedBinding) -> WrappedDek
```

The service resolves the `KeyRef` server-side from the authenticated tenant,
the purpose, and tenant policy. A caller never selects a provider or
supplies a `KeyRef` on this path; the reference returned to the port is a
service-owned argument produced after that resolution, not an authoritative
caller choice
([ADR#0030](../adr/0030-customer-controlled-key-backend-routing.md)
Decision 1).

### unwrap_dek

```text
unwrap_dek(KeyRef, WrappedDek, AuthenticatedBinding) -> Dek
```

A caller may return the opaque `KeyRef` stored alongside an envelope, but the
service treats it as untrusted input. It verifies that the `KeyRef`'s bound
tenant and purpose match the authenticated operation before invoking an
adapter. A mismatch fails as `BindingMismatch`, never as a silent fallback.

### BindExternalKey

An authorized control-plane operation, never a data-plane one. After a
customer provisions a key in their own provider, `BindExternalKey` accepts
one `KeyBackendId`, one exact native locator, the key's purpose, and its
required capabilities. The service validates provider identity, access, and
a complete authenticated-data round trip, then returns a new opaque `KeyRef`.
Provider and locator selection are allowed only here, never on `wrap_dek` or
`unwrap_dek`
([ADR#0030](../adr/0030-customer-controlled-key-backend-routing.md)
Decision 4).

### Rewrap migration

Moving ciphertext to another backend is an explicit migration, not a
configuration cutover. It creates a new `KeyRef` bound to the new backend,
unwraps under the old `KeyRef`, wraps under the new `KeyRef`, and commits
the new reference and `WrappedDek` as one transactional or compare-and-set
mutation against the envelope's expected version, recording a stable
migration operation ID. If provider wrapping succeeds but the mutation
conflicts, no partial pair is committed, and the currently committed pair
remains authoritative. Both backends must be available for the duration of
the migration, and there is no fallback from one backend to the other.

This rewrap flow applies only to a custody migration when the old
key-encryption key is not compromised. A suspected compromise requires new
DEKs and payload re-encryption under
[ADR#0023](../adr/0023-secret-management-and-key-custody-direction.md), not
reuse of the existing DEKs
([ADR#0030](../adr/0030-customer-controlled-key-backend-routing.md)
Decision 4).

## Custody tiers

`KeyManagement` backends are provider adapters, not customer-facing tiers.
The product offers exactly two tiers, [managed key](../glossary/managed-key)
and [customer managed key](../glossary/customer-managed-key), and tier is a
question of who owns the backend registration, never which vendor
implements it. See
[ADR#0033](../adr/0033-two-tier-key-custody-product-model.md) and
[Key Custody](./key-custody.md) for the full model and the reasoning behind
it.

| Backend | Tier |
| --- | --- |
| Platform-operated OpenBao Transit | Managed key |
| Customer AWS KMS key | Customer managed key |
| Customer Google Cloud KMS key | Customer managed key |
| Customer-operated OpenBao Transit | Customer managed key |

## Failure handling

Every `wrap_dek` and `unwrap_dek` outcome maps to one of eight stable domain
failure categories
([ADR#0030](../adr/0030-customer-controlled-key-backend-routing.md)
Decision 6). `BindExternalKey` has its own rejection rules, described under
[BindExternalKey](#bindexternalkey)
([ADR#0030](../adr/0030-customer-controlled-key-backend-routing.md)
Decision 7):

| Category | Includes | Automatic retry | Audit classification |
| --- | --- | --- | --- |
| `KeyUseDenied` | Customer authorization denial, or a disabled, destroyed, or missing key | No; external customer action must change first | Security denial |
| `ProviderUnavailable` | Transport failure, timeout, or provider service failure | Bounded backoff within the operation deadline | Operational failure |
| `ProviderThrottled` | Explicit provider quota or rate limit | Bounded provider-directed backoff | Capacity failure |
| `BindingMismatch` | `KeyRef` tenant or purpose does not match the authenticated operation | No | Security violation |
| `TransportIntegrityMismatch` | Provider request checksum rejection or response checksum mismatch | Limited same-provider retry; persistent mismatch is terminal | Transport integrity violation |
| `IntegrityFailure` | Cryptographic authenticated-data or ciphertext validation fails | No | Cryptographic integrity violation |
| `UnsupportedCapability` | The bound key lacks the requested typed capability | No | Contract rejection |
| `InternalFailure` | Unmapped provider error, malformed response, or adapter invariant violation | No | Adapter fault |

Standing rules that hold across all eight categories:

- The service fails closed. No wrap or unwrap succeeds without participation
  by the backend bound to the `KeyRef`, and a failure never produces a
  successful degraded mode.
- There is no fallback between backends, at any category, for any tenant.
- A retry never switches providers; a `ProviderUnavailable` or
  `ProviderThrottled` retry stays bounded and stays on the same backend.
- A public response may coarsen denied, missing, and disabled outcomes to
  avoid giving an unauthenticated or cross-tenant caller a resource oracle,
  while the internal category and audit record retain the full detail.

## What the service never does

- Retain a usable, unwrapped `Dek` in application state or a cache after its
  in-flight operation completes.
- Pre-unwrap a `Dek` for later use.
- Fall back to a platform key when a customer-controlled backend denies or
  fails.
- Let a provider artifact (a path, a token, a policy name, a ciphertext
  version marker, an SDK type) cross the service boundary outside the opaque
  payload carried inside a `WrappedDek`.
- Accept backend or locator selection from a caller on `wrap_dek` or
  `unwrap_dek`.

## Open decisions

- **Versioned envelope format and payload cipher suite.** The complete
  format, the payload cipher, and the nonce construction are not yet chosen,
  and that decision must exist before adapters are built. The `Dek` size this
  page states is a target for the initial conformance suite, not a
  ratified value, and is validated only once the cipher suite is chosen
  ([ADR#0030](../adr/0030-customer-controlled-key-backend-routing.md)
  Non-Goals).
- **Scheduled destruction waiting period.** [ADR#0023](../adr/0023-secret-management-and-key-custody-direction.md) fixes that destruction
  of a platform-managed key is scheduled and reviewable rather than
  immediate, but does not fix how long the waiting period is
  ([ADR#0023](../adr/0023-secret-management-and-key-custody-direction.md)
  Decision 6).
- **Console and API request shapes.** Neither [ADR#0030](../adr/0030-customer-controlled-key-backend-routing.md) nor [ADR#0033](../adr/0033-two-tier-key-custody-product-model.md) defines
  console screens or API request and response shapes; both fix only the
  vocabulary and boundaries those surfaces must respect
  ([ADR#0033](../adr/0033-two-tier-key-custody-product-model.md)
  Non-Goals).
